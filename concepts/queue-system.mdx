When you need a sequence of animations to play back-to-back — such as a multi-hit attack combo or a cutscene gesture chain — manually listening for each track to finish and then triggering the next one adds significant boilerplate. EZAnim's queue system handles this automatically: you push animation IDs into a per-model FIFO list and each one plays the moment the previous non-looping track stops.

## How the Queue Works

Every model has its own queue array. `queue(model, id)` appends an animation ID to the end of that array. When a non-looping animation finishes, EZAnim automatically picks the first ID off the front of the queue and plays it. This continues until the queue is empty.

```lua
-- Play an attack combo
Anim:play(character, "Attack1", "Action", false)
Anim:queue(character, "Attack2")
Anim:queue(character, "Attack3")
-- Attack2 plays when Attack1 ends, then Attack3 plays when Attack2 ends
```

You can queue any number of IDs before the first animation even finishes — they will drain in order.

## Non-Looping Requirement

The queue only advances when a non-looping animation stops. If you call `play()` with `looped = true`, the track will never naturally end on its own, so the queue will never drain. Always use `looped = false` for animations that are part of a queue sequence.

```lua
-- Correct: non-looping animations advance the queue
Anim:play(character, "Attack1", "Action", false)
Anim:queue(character, "Attack2")

-- Incorrect: looping animation blocks the queue indefinitely
Anim:play(character, "Attack1", "Action", true)  -- queue never drains
Anim:queue(character, "Attack2")
```

## Clearing the Queue

To cancel all pending queued animations and stop everything currently playing, call `reset()`:

```lua
-- Cancel the rest of the queue and stop all animations
Anim:reset(character)
```

This is useful when an external event — such as the character taking damage or dying — should immediately abort a combo in progress and transition to a different animation.

## Returning to Idle After a Combo

A common pattern is to return a character to their idle animation once an entire queued sequence completes. Because the queue drains automatically and each animation plays non-looped, you can use `setState` and `getState` to track when the combo is finished and conditionally play the idle from your game logic:

```lua
Anim:setState(character, "Combo")
Anim:play(character, "Attack1", "Action", false)
Anim:queue(character, "Attack2")
Anim:queue(character, "Attack3")

-- In your game loop or event, check when the combo state ends
-- and return to idle once all queued animations have completed
```

<Tip>
  Combine the queue system with your own state flags (see the [State System](/concepts/state-system) page) to guard against new inputs while a combo is in progress, and to cleanly return to idle once the queue drains.
</Tip>

<Note>
  Queued animations are played with default parameters: `Movement` priority, non-looping, and a 0.15-second fade time. If you need a different priority, a specific fade duration, or looping behaviour for any animation in the sequence, call `play()` directly and chain your own logic — the queue is best suited for simple, uniform sequences.
</Note>
